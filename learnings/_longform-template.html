Server-Sent Events

Verified Intermediate ~7 min read Used in · STATUS stream 2026-06-05

A long-lived HTTP connection that streams text events from server to client. It delivers one-way push over standard HTTP, with automatic reconnection.

01Mental model

One connection, many messages

A standard request completes once its body is sent. SSE instead holds the response open and emits events as they occur, each one a data: line followed by a blank line. The browser's EventSource parses each event and dispatches a message. A single connection carries many messages until either party closes it.

02Why we use it here

A read-only channel

The live STATUS dashboard reflects git activity and ticket transitions as agents write them. The client never sends data back over this channel, so a bidirectional transport such as WebSocket would add operational overhead without a corresponding benefit. SSE provides the server-to-client stream over standard HTTP, which existing proxies handle without special configuration.

03How it works
client.jsEventSource
const es = new EventSource('/status/stream');
es.onmessage = (e) => {
  const update = JSON.parse(e.data);
  render(update);            // repaint dashboard
};
es.addEventListener('done', () => es.close());
server.jsNode · zero deps
res.writeHead(200, {
  'Content-Type': 'text/event-stream',
  'Cache-Control': 'no-cache',
  'Connection': 'keep-alive',

const tick = () => res.write(`data: ${JSON.stringify(state)}\n\n`);

06Constraints & trade-offs
  1. The channel is server-to-client only; client requests use a separate fetch() / POST.
  2. HTTP/1.1 limits a browser to roughly six concurrent connections per origin; HTTP/2 multiplexing removes the constraint.
  3. Intermediaries may buffer the response; disable buffering and emit periodic heartbeats.
  4. The protocol carries UTF-8 text only; binary payloads must be base64-encoded.
