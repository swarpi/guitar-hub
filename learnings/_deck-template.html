Server-Sent Events

A long-lived HTTP connection that streams text events from server to client. It delivers one-way push over standard HTTP, with automatic reconnection.
Verified Intermediate ~7 min read Used in · STATUS stream

What the dashboard needs

01

Server-initiated

Updates come from agent writes, not user actions.

02

One-way

The browser consumes events; it never replies on this channel.

03

Near real-time

A write should reach the screen within about a second.

04

Self-healing

A dropped connection reconnects on its own.

06At scale
01

One viewer, one connection

10k dashboards means 10k long-lived connections to plan for.

02

Per-origin cap

HTTP/1.1 allows ~6 per origin; HTTP/2 multiplexes ~100 on one.

03

Heartbeats

A comment line every ~30s stops proxies closing an idle stream.

04

Text only

UTF-8 payloads; encode binary as base64.

—Wrap up

When to choose SSE

SSE fits server-to-client streams where updates flow one way and standard HTTP compatibility matters. The choice comes down to direction and volume.
Use SSE
Live dashboards, activity feeds, log tails, notifications.
Use WebSocket
The client must also send a steady stream of messages.
Just poll
Only a few updates per minute; simpler and good enough.
