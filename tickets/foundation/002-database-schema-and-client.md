# Ticket: Database Schema and Drizzle Client

**Feature:** foundation
**Status:** Done
**Priority:** P0
**Estimate:** M
**Related:** ADR-0002

## Context

ADR-0002 specifies two tables â€” `artists` and `songs` â€” backed by Cloudflare D1 (SQLite). The current prototype stores data in `localStorage` via `src/lib/store.ts`, which uses a flat `Song` interface with an embedded `artist` string. The ADR's schema normalizes artists into their own table with a foreign key relationship. This ticket creates the Drizzle schema, the database client factory, and the SQL migration file so that subsequent tickets can write server actions against a real database.

## Goal

Define the Drizzle schema for `artists` and `songs`, generate the initial SQL migration, and expose a typed database client that server actions can import.

## Notes

- Drizzle schema for SQLite uses `sqliteTable` from `drizzle-orm/sqlite-core`. Column types: `text()`, `integer()`, `integer({ mode: "number" })`.
- ISO 8601 timestamps are stored as `text()`. Use `.$defaultFn(() => new Date().toISOString())` for `created_at` and `updated_at`.
- The `capo` column is `integer().notNull(false)` â€” Drizzle SQLite integers are nullable by default unless `.notNull()` is chained.
- For D1 in Next.js App Router, the `env` object is accessed via `getRequestContext().env` from `@cloudflare/next-on-pages`. The client factory wraps this pattern.
- The Drizzle test can use `drizzle(new Database(':memory:'))` with `better-sqlite3` for unit testing the schema without a real D1 instance. Add `better-sqlite3` and `@types/better-sqlite3` as dev dependencies in this ticket.
